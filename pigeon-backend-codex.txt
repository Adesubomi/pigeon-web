1. REQUIRED MODELS
------------------

  id: string
  name: string
  url: string
  status: "active" | "inactive"

  path: string
  time: string
  receivedAt: string
  size: string

  payload: string
  headers: Record<string, string>

  endpointId: string
  name: string
  localUrl: string
  enabled: boolean

------------------------

  Response:

      user: { id, name, email, avatarUrl?, timezone },
      workspace: { id, name, role },

  Used by: sign-in screen "Continue with Google".
  Starts Google OAuth.

  Used by: sign-in screen "Continue with GitHub".
  Starts GitHub OAuth.

GET /api/auth/:provider/callback
  Used by: OAuth providers.
  Completes OAuth and establishes the app session.

POST /api/auth/logout
  Used by: app shell when sign out is added.
  Ends the session.

6. PUBLIC WEBHOOK INGESTION
---------------------------

ANY /e/:endpoint_id/:path*
  Used by: external webhook senders.
  Accepts: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS.

ANY /:path* on endpoint subdomains
  Used by: production relay URLs when the endpoint id/subdomain is in the hostname.
  Example production shape:
    https://{endpoint_id}.pigeon.sh/orders/create

Webhook ingest behavior:
  - Resolve the endpoint from path or hostname.
  - Verify endpoint exists.
  - Apply endpoint active/inactive policy.
  - Store raw request data:
      method,
      path,
      query,
      headers,
      raw body,
      source IP,
      size,
      receivedAt
  - Create an EndpointEvent.
  - Push the new event to inspection sockets.
  - Relay the request to enabled connected clients.
  - Return quickly to the webhook sender.

Suggested ingest responses:
  200 { ok: true, eventId: string }
  204 for HEAD or empty OPTIONS handling
  404 for unknown endpoint
  410 if paused endpoints should reject traffic
  413 for payloads over plan limit
  429 for rate limits


7. INSPECTION SCREEN HANDSHAKE AND UPGRADE
------------------------------------------

Frontend routes:
  GET /e/inspect/:endpoint_id
  GET /e/inspect/:endpoint_id/events/:event_id

These are page routes in Nuxt. The backend still needs data and a realtime stream for the inspection screen.

GET /api/inspect/:endpoint_id
  Used by: inspection screen initial render.
  Returns:
    {
      endpoint: Endpoint,
      connected: boolean,
      events: EndpointEvent[]
    }

GET /api/inspect/:endpoint_id/events/:event_id
  Used by: selected inspection event detail.
  Returns: EndpointEvent

GET /ws/inspect/:endpoint_id
  Used by: inspection screen realtime updates.
  This endpoint performs the HTTP webhook/live-inspection handshake, then upgrades to WebSocket.

Handshake request:
  GET /ws/inspect/:endpoint_id HTTP/1.1
  Host: relay.pigeon.sh
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Key: <base64 client nonce>
  Sec-WebSocket-Version: 13
  Sec-WebSocket-Protocol: pigeon.inspect.v1
  Cookie: pigeon_session=<session cookie>

Successful upgrade response:
  HTTP/1.1 101 Switching Protocols
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Accept: <derived accept key>
  Sec-WebSocket-Protocol: pigeon.inspect.v1

Server messages after upgrade:
  {
    "type": "snapshot",
    "endpoint": Endpoint,
    "connected": boolean,
    "events": EndpointEvent[]
  }

  {
    "type": "event.created",
    "event": EndpointEvent
  }

  {
    "type": "endpoint.status",
    "status": "active" | "inactive",
    "connected": boolean,
    "deviceCount": number
  }

  {
    "type": "client.connected",
    "client": ConnectedClient
  }

  {
    "type": "client.disconnected",
    "clientId": string
  }

  {
    "type": "pong",
    "ts": string
  }

Browser messages after upgrade:
  {
    "type": "subscribe",
    "endpointId": string
  }

  {
    "type": "ping",
    "ts": string
  }

Inspection behavior required by the UI:
  - Shows "Connected" when endpoint is active and device count is above zero.
  - Shows incoming events live.
  - Opens event detail by event id.
  - Event detail exposes body, headers, method, path, received timestamp, size, and source.

Optional fallback:

GET /sse/inspect/:endpoint_id
  Used when WebSockets are blocked.
  Content-Type: text/event-stream
  Event names: snapshot, event.created, endpoint.status, client.connected, client.disconnected.


8. CONNECTED CLIENTS AND LOCAL RELAY UPGRADE
--------------------------------------------

GET /api/endpoints/:endpoint_id/clients
  Used by: endpoint Clients tab.
  Returns: ConnectedClient[]

POST /api/endpoints/:endpoint_id/clients/pairing
  Used by: "Connect client".
  Returns:
    {
      endpointId: string,
      pairingCode: string,
      expiresAt: string
    }
  UI expects pairingCode shaped like XXXX-XXXX-XXXX.

PATCH /api/endpoints/:endpoint_id/clients/:client_id
  Used by: client enable/disable switch.
  Body: { enabled: boolean }
  Returns:
    {
      endpointId: string,
      clientId: string,
      enabled: boolean
    }

DELETE /api/endpoints/:endpoint_id/clients/:client_id
  Used by: delete client button.
  Returns:
    {
      endpointId: string,
      clientId: string
    }

GET /ws/clients
  Used by: local relay client app/CLI.
  This endpoint performs the client handshake and upgrades to WebSocket.

Handshake request:
  GET /ws/clients HTTP/1.1
  Host: relay.pigeon.sh
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Key: <base64 client nonce>
  Sec-WebSocket-Version: 13
  Sec-WebSocket-Protocol: pigeon.client.v1

Successful upgrade response:
  HTTP/1.1 101 Switching Protocols
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Accept: <derived accept key>
  Sec-WebSocket-Protocol: pigeon.client.v1

First message from new client:
  {
    "type": "pair",
    "pairingCode": "XXXX-XXXX-XXXX",
    "client": {
      "name": string,
      "localUrl": string
    }
  }

Server response to new client:
  {
    "type": "paired",
    "endpointId": string,
    "client": ConnectedClient,
    "token": string
  }

First message from returning client:
  {
    "type": "authenticate",
    "clientId": string,
    "token": string
  }

Server-to-client delivery message:
  {
    "type": "webhook.request",
    "deliveryId": string,
    "event": EndpointEvent,
    "target": {
      "localUrl": string
    }
  }

Client-to-server delivery result:
  {
    "type": "webhook.response",
    "deliveryId": string,
    "status": number,
    "headers": Record<string, string>,
    "body": string,
    "durationMs": number
  }

Client behavior required by endpoint Clients tab:
  - Pairing code starts in waiting state.
  - A successful websocket pair should create a ConnectedClient.
  - Inspection and endpoint detail screens should receive client.connected.
  - Disabled clients remain listed but do not receive webhook.request deliveries.

11. ERROR CODES TO STANDARDIZE
------------------------------

400 Bad Request
  Malformed JSON, missing required fields, unsupported method shape.

401 Unauthorized
  Missing or expired dashboard session.

403 Forbidden
  User does not belong to the workspace or cannot access the resource.

404 Not Found
  Unknown endpoint, event, client, invoice, account, or provider.

409 Conflict
  Duplicate subdomain, duplicate pairing, already-connected provider.

410 Gone
  Deleted endpoint or intentionally unavailable relay.

413 Payload Too Large
  Webhook payload exceeds workspace plan limit.

422 Unprocessable Entity
  Invalid subdomain, invalid pairing code, invalid downgrade/support form.

429 Too Many Requests
  Rate limit for webhook ingestion, replay, pairing, auth, or billing actions.
